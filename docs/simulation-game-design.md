# 『HEARTHFALL（仮称）』設計図 — 創発的ドラマを生むシミュレーションゲーム

> 設計思想: プレイヤーの選択が「生態系 × 経済 × 心理」の3層を貫いてフィードバックし、
> 開発者が書いていない物語が"勝手に"立ち上がる。これを技術的に再現可能なレベルで設計する。

本書は単なるコンセプト集ではなく、各システムを**数式・データ構造・アルゴリズム**まで落とし込み、
そのまま実装タスクに分解できる粒度で記述する。各節末に「なぜこの設計が良いか（理論的根拠）」を付す。

---

## 0. 全体像 — なぜ「3層フィードバックループ」なのか

創発（emergence）は「単純な規則の局所的相互作用が、設計されていない大域的秩序を生む」現象である。
Conway's Life、Boids、Dwarf Fortress いずれも共通するのは **(1) 構成要素が状態を持ち、(2) 近傍と相互作用し、(3) その結果が次の入力に戻る（閉ループ）** という構造だ。

本ゲームは3つの相互依存する動的システムを「閉ループ」で結合する。

```
        ┌──────────────┐  資源収量/災害   ┌──────────────┐
        │  ECOLOGY 層   │ ───────────────▶ │  ECONOMY 層   │
        │ 気候/土壌/動植物│ ◀─────────────── │ 生産/交易/価格 │
        └──────┬───────┘   伐採/汚染/開墾   └──────┬───────┘
               │ 飢餓/天災/美観              欠乏/富/労働 │
               ▼                                    ▼
        ┌──────────────────────────────────────────────┐
        │            PSYCHE 層（住人の感情・関係・記憶）       │
        │  個々の意思決定が ECOLOGY と ECONOMY に作用を返す    │
        └──────────────────────────────────────────────┘
```

**設計原則**: プレイヤーは「層に直接命令する」のではなく「層の境界条件（インセンティブ・制約）を操作する」。
神の手で個人を動かすのではなく、税率・法・インフラ・ゾーニングといった**場のパラメータ**を変える。
この「間接統治」が予測不可能性の源泉になる（→ 2-3 で詳述）。

## Step 1: コンセプトの定義

### 1.1 コアサイクル（最も楽しい反復行動）

コアサイクルは **「観察 → 仮説 → 介入 → 波及の観測 → 再介入」** の5フェーズループとする。

| フェーズ | プレイヤー行動 | 時間スケール | 提供する快感 |
|---|---|---|---|
| 観察 (Observe) | オーバーレイで異常・偏り・関係を読む | 秒〜分 | 発見（"なぜ西区の食料が腐る？"） |
| 仮説 (Hypothesize) | 因果を推論する | 秒 | 知的興奮 |
| 介入 (Intervene) | 場のパラメータを操作（道を引く/税を変える/法を制定） | 秒 | 主体性 |
| 波及 (Ripple) | シミュレーションが数十〜数百tick伝播 | 分〜時 | 期待と緊張 |
| 観測 (Reflect) | 意図した結果＋**意図しない副作用**を見る | 分 | 驚き・物語 |

このループの肝は **「介入の即時性」と「結果の遅延性」の落差** にある。
介入は1クリックで済む（低コスト・高頻度）が、結果は時間をかけて非線形に広がる（高インパクト・予測困難）。
この非対称性が「もう1回介入したい」という中毒性を生む。

**理論的根拠**: 行動心理学の「変動比率強化スケジュール（variable-ratio reinforcement）」。
介入に対する報酬（良い結果）が確率的かつ可変的に返るほど、行動は強化され持続する。
完全に予測可能なシステム（決定論的すぎる）は飽き、完全にランダムなシステムは無力感を生む。
**「7割は読めるが3割は読めない」** という予測可能性のスイートスポットを狙う（→ 2.4 のカオス調整で実現）。

### 1.2 カタルシス（達成感の瞬間）

達成感を3階層で設計する。スケールの異なる報酬を重ねることで、短期・中期・長期の動機を同時に維持する。

1. **マイクロ・カタルシス（数分）**: 詰まっていた物流ボトルネックが解消し、倉庫が満ちる瞬間。
   視覚（流れの開通）と聴覚（達成音）で即時フィードバック。
2. **メゾ・カタルシス（数時間）**: 対立していた2人の住人が、プレイヤーの作った共有空間がきっかけで和解し婚姻する。
   → これは**スクリプトではなく**関係グラフと記憶システムの帰結として発生する（→ 2.2.3）。
3. **マクロ・カタルシス（数十時間）**: 飢饉・疫病・内紛を乗り越えた集落が世代交代し、
   初代住人の孫が「祖父の物語」を語る。プレイヤーは自分の介入の累積が「歴史」になったことを実感する。

**最重要カタルシス＝「Anticipated Disaster Averted（予見した災厄の回避）」**:
プレイヤーが因果を読み切り、まだ顕在化していない危機（例: 単作による土壌疲弊→次の冬の飢饉）を
事前の介入で防いだとき。Factorio/RimWorld が共通して持つ最高の瞬間であり、
これは **「自分だけがシステムを理解している」という全能感** を与える。

**理論的根拠**: カタルシスの強度は「結果の重大さ × プレイヤー自身の因果寄与の明確さ」に比例する。
だからこそ §4 のUIは「自分の介入が何を引き起こしたか」を**追跡可能（traceable）**にせねばならない。
寄与が不明瞭な成功は運に見え、カタルシスにならない。

## Step 2: システムデザイン（複雑系）

### 2.1 ECOLOGY 層 — 拡散方程式ベースの環境シミュレーション

世界をヘクスまたは正方グリッドの**タイル**で表現し、各タイルが状態ベクトルを持つ。

```
TileState = {
  soil_fertility:  f32 ∈ [0,1]   // 土壌肥沃度
  moisture:        f32 ∈ [0,1]   // 水分
  temperature:     f32           // 摂氏
  pollution:       f32 ∈ [0,1]   // 汚染
  biomass:         f32           // 植生量
  elevation:       f32           // 標高（不変）
}
```

**水分・汚染・温度の伝播は離散化した拡散方程式（熱方程式の数値解法）で計算する:**

```
∂φ/∂t = D ∇²φ + S - L

離散形（4近傍ラプラシアン）:
φ_i(t+1) = φ_i(t) + D·Σ_{j∈N(i)} (φ_j(t) − φ_i(t)) + Source_i − Loss_i

  D      : 拡散係数（汚染は速く、土壌養分は遅い、と物質ごとに変える）
  N(i)   : タイル i の近傍
  Source : 雨/工場排出など
  Loss   : 蒸発/自然分解
```

**植生・動物は別レイヤーのエージェント**で、ロトカ＝ヴォルテラ型の捕食被食モデルを用いる:

```
dPrey/dt  = a·Prey − b·Prey·Pred       // 草食動物
dPred/dt  = c·Prey·Pred − d·Pred        // 捕食動物
（タイル局所で離散積分。biomass が a を変調 ＝ 餌が多いと繁殖）
```

これにより「狼を獲りすぎる→鹿が増える→草を食い尽くす→翌年鹿が餓死」という
**設計されていない生態カスケード**が自然に発生する。

> **なぜ拡散方程式か**: 汚染や水を「イベント」で扱うと境界が不自然になり、プレイヤーの空間的直観が裏切られる。
> 連続場として扱えば「川の上流に工場を置くと下流の農地が汚れる」という**空間的因果がプレイヤーに学習可能**になる。
> 学習可能な物理こそが「仮説→検証」のコアサイクルを成立させる。

### 2.2 PSYCHE 層 — 住人の感情・記憶・関係

創発的ドラマの心臓部。住人（Colonist）は以下を持つ。

#### 2.2.1 感情モデル: PAD空間 + 気分（Mood）

感情を離散ラベル（嬉しい/悲しい）ではなく **PAD連続3次元**で表す。

```
PAD = (Pleasure, Arousal, Dominance) ∈ [−1,1]³
  Pleasure : 快/不快
  Arousal  : 覚醒/沈静
  Dominance: 支配/服従感
```

イベントは PAD への**インパルス**を与え、感情はホメオスタシス（基準点）へ指数減衰で戻る:

```
PAD(t+1) = PAD(t) + Σ event_impulse − k·(PAD(t) − baseline_personality)

baseline_personality は性格（後述 Big Five）で個体差を持つ
```

PAD ラベルへの写像例: 高 P・高 A・高 D = 「歓喜・誇り」、低 P・高 A・低 D =「恐怖・パニック」。
これにより同じ事件でも**性格と文脈で反応が変わる**（決定論的セリフ分岐の排除）。

#### 2.2.2 ニーズと意思決定: Utility AI（効用ベースAI）

各住人は毎tick、候補行動の**効用スコア**を計算し最大のものを選ぶ（GOAPより軽量で創発的）。

```
Utility(action) = Σ_n  w_n · Curve_n( Need_n )  ·  Context(action)

例: Need = {hunger, sleep, social, comfort, safety, esteem, purpose}
Curve は Need 充足度を効用に写すレスポンスカーブ（飢餓は指数、社交は線形 等）
Context = 距離・所有・関係・現在の感情による補正
```

行動選択は決定論だが、入力（ニーズ・文脈・関係）が高次元なため**実質的に予測不能な多様性**が出る。

> **なぜ Utility AI か**: 有限状態機械（FSM）は状態爆発し、振る舞いが「読めて」しまう。
> ビヘイビアツリーは設計者の意図が透ける。Utility AI は **「複数の欲求の重み付き競合」** を表現でき、
> 同じ住人が空腹と社交欲の僅差で日替わりに違う選択をする ＝ 生きている錯覚を生む。
> しかも各カーブはデータ駆動で**MODやバランス調整が容易**。

#### 2.2.3 関係グラフと記憶

住人間の関係を**重み付き有向グラフ** G=(V,E) で保持する。

```
RelationshipEdge(A→B) = {
  affinity:  f32 ∈ [−1,1]   // 好意/敵意
  trust:     f32 ∈ [−1,1]
  familiarity: f32 ∈ [0,1]  // 接触量
  tags: Set<RelType>        // family / lover / rival / mentor ...
}
```

各住人は**記憶（Memory）の時系列ログ**を持つ。記憶は重要度で減衰し、想起されると感情に再作用する。

```
Memory = { event_ref, salience: f32, valence: f32, timestamp, subjects: [EntityId] }
salience(t) = salience₀ · exp(−λ·age) · (1 + rehearsal_count·β)
  // Ebbinghaus 忘却曲線 + 反芻による強化
```

**ドラマ生成の連鎖（例: 設計されていない復讐劇）:**
1. 飢饉で食料を巡り住人 X が Y を突き飛ばす（ECONOMY層の欠乏が原因）
2. Y に高 salience・負 valence の記憶が刻まれ、G(Y→X).affinity が急落
3. Y の Utility AI が X 関連行動にネガティブ補正、回避・敵対行動の効用が上昇
4. 後日、X が事故で負傷 → Y は「助ける」効用が低く見殺し気味の選択
5. 目撃した第三者 Z の道徳値により Z→Y 関係が悪化…と**波及が連鎖**

この連鎖は1行もイベントスクリプトを書いていない。**層間結合の必然的帰結**である。

#### 2.2.4 性格: Big Five（OCEAN）

```
Personality = {openness, conscientiousness, extraversion, agreeableness, neuroticism} ∈ [0,1]⁵
```

性格は (a) PAD baseline、(b) Utility 重み、(c) 記憶の減衰率（neuroticism 高＝負記憶が残る）を変調する。
たった5次元で住人に**一貫した個性**を与えられる（心理学的妥当性 ＝ プレイヤーが「人間らしさ」を感じる）。

### 2.3 ECONOMY 層 — エージェントベースの市場

中央計画の価格表を**置かない**。価格は需給から**創発させる**。

```
各 GoodType ごとにローカル市場を持ち、注文板（order book）で価格発見:
  住人/工房は「売り注文」「買い注文」を効用と在庫から生成
  約定価格 P は需給均衡で決まる:
    P(t+1) = P(t) · (1 + κ·(Demand − Supply)/(Demand + Supply + ε))
  輸送は graph 上の最短/最安経路（交易路ネットワーク）で価格をアービトラージ
```

生産は**有向非巡回グラフ（DAG）の技術ツリー**で表す（Factorio型）。
ただしレシピは固定だが、**何を作るかは市場価格に応じて住人/工房が自律選択**する。

> **なぜ創発的価格か**: 固定価格経済は「最適解」が一意に定まり、解けばゲームが終わる。
> 需給創発価格は外的ショック（不作・人口流入）で常に揺れ、**恒常的な最適解が存在しない**。
> これが「終わらない面白さ（infinite replayability）」を保証する。Dwarf Fortress 的な無限の物語性の経済版。

### 2.4 カオスと予測可能性の調整 — 「7:3の法則」の実装

純粋な決定論は飽き、純粋なランダムは理不尽。両者を**構造的に**混ぜる。

```
結果のばらつき = 決定論的コア（読める）＋ 有界ノイズ（読めない）
  - 個々の住人選択: 決定論（同入力→同出力）。再現性とデバッグ性を担保
  - 大域イベント（天候・隕石・疫病）: シード付き擬似乱数 ＋ 状態依存ハザード関数
      P(disaster | state) = base_rate · stress_multiplier(state)
      // 過密・汚染・単作など「脆弱な状態」ほど災害確率が上がる
```

ポイントは **「ランダムは状態依存で重み付けされる」** こと。
プレイヤーの悪手（汚染放置）が確率を上げるので、**運ではなく因果として体験される**。
これがカタルシス（§1.2）の「因果寄与の明確さ」を担保する。

### 2.5 プレイヤー操作 vs 自動シミュレーションの境界線

**設計の中核哲学: プレイヤーは「ミクロを操作せず、マクロの場を整える」。**

| 領域 | 主体 | 具体例 |
|---|---|---|
| **直接操作（プレイヤー）** | 場・制約・インセンティブ | ゾーニング、道路/水路敷設、法・税・配給ルール、建設許可、優先度ポリシー、研究投資 |
| **半自動（ポリシー経由の間接操作）** | プレイヤーが規則→AIが実行 | 「冬は食料備蓄を優先」等の宣言的ルール。プレイヤーは*意図*を、AIが*手段*を担う |
| **完全自動（シミュレーション）** | システム | 個々の住人の毎tick行動、感情、関係、価格、生態、天候 |

**境界の原則「Authoring vs Simulating」**:
プレイヤーが操作するのは**変更頻度が低く影響範囲が広い**もの（法・インフラ）。
シミュレーションが担うのは**変更頻度が高く局所的**なもの（誰が今どこで何を食べるか）。

> **なぜこの境界か**: もしプレイヤーが個々の住人を直接操作（RTSのユニット選択）すると、
> マイクロ管理地獄になり、かつ住人の「自律性＝生命感」が消える。創発が死ぬ。
> 逆に何も操作できないと放置ゲームになる。
> 「場を整え、自律エージェントがその場で踊るのを見る」 ＝ **庭師（gardener）のメタファー**。
> SimCity の市長、RimWorld の総督、Factorio の自動化設計者は皆この立ち位置にいる。これが王道かつ最適。

## Step 3: 技術アーキテクチャ

### 3.1 中核: ECS（Entity Component System）+ Data-Oriented Design

何万体もの住人・タイル・物品を毎tick更新するため、OOPの継承ツリーではなく **ECS** を採用する。

```
Entity     : ただのID（u64）。データもロジックも持たない
Component  : 純粋データ（構造体）。SoA（Structure of Arrays）で連続メモリ配置
System     : Component集合に対して一括処理する関数

例:
  Components: Position, NeedsBundle, PAD, Personality, Inventory,
              RelationshipRefs, MemoryLog, JobAssignment, TileState ...
  Systems:    NeedDecaySystem → UtilityDecisionSystem → MovementSystem →
              InteractionSystem → EmotionUpdateSystem → EconomyMatchSystem →
              EcologyDiffusionSystem → DisasterHazardSystem → ChronicleSystem
```

> **なぜ ECS / DOD か**:
> 1. **キャッシュ効率**: 同種Componentが連続配置されるため、CPUキャッシュミスが激減（OOPの10倍級の差が出る場面も）。
>    1万エージェント × 60tick/秒 にはこれが必須。
> 2. **並列化**: 各SystemはアクセスするComponentが明示的なので、依存のないSystemを**データ並列**で実行可能。
> 3. **合成（composition over inheritance）**: 「翼を持つ家畜」「病気の商人」を継承爆発なしに表現でき、MOD拡張に強い。
> 推奨実装: Rust + `bevy_ecs` / `flecs`、または C++ + EnTT。決定論が要るのでECSのシステム実行順は固定する。

### 3.2 性能: 階層的時間分割（LOD Simulation / Time-Slicing）

全エンティティを毎tickフル計算するのは無駄。**シミュレーションLOD**を導入する。

```
LOD0 (画面内・注目中): 毎tick フル計算（感情・関係・詳細行動）
LOD1 (近傍・非表示)  : 数tickに1回。行動を粗い統計で近似
LOD2 (遠方)          : 集計シミュレーション（個を捨て、人口・資源を集団方程式で更新）
        例: 遠方の村は個々の住人を消し、人口×平均生産で経済だけ回す
        プレイヤーが近づいた瞬間、統計から個体を再インスタンス化（reify）
```

加えて重い処理は**バジェット制**で分散:

```
毎tickの計算予算（例 8ms）を超えたら、未処理エンティティを次tickへ繰り越す
  → 経路探索・関係再評価などは「ラウンドロビンで毎フレーム一部だけ」処理
```

> **なぜ LOD か**: Dwarf Fortress の有名な失速は全エンティティ完全計算が一因。
> 知覚的に重要な対象（プレイヤーが見ているもの）だけ精密に、それ以外は安価な近似で回せば、
> **体感品質を落とさず10〜100倍スケールできる**。「人は見ていない所の精度を知覚できない」を逆手に取る。

### 3.3 空間: 空間分割とグラフ

- **空間クエリ**（近傍探索・拡散・視界）: 一様グリッド or 四分木（quadtree）。
  タイルベースなのでグリッドのO(1)近傍が基本。
- **経路探索**: タイル上で **HPA*（Hierarchical Pathfinding A*）**。
  マップをクラスタに分割し、クラスタ間グラフで粗探索→クラスタ内で詳細探索。
  1万体の同時経路探索を現実的コストにする。**フローフィールド**を群衆移動に併用。
- **交易・関係・技術ツリー**: 明示的グラフ構造。交易はネットワーク上の最小費用流、
  技術ツリーはDAGのトポロジカル制約、関係は前述の重み付き有向グラフ。

> **なぜグラフ理論か**: 経済の伝播、噂・疫病の拡散、関係の連鎖はすべて
> 「ノード間を伝わる量」のグラフ問題に帰着する。同一の抽象（グラフ走査・最短路・連結成分）で
> 複数システムを統一実装でき、**コード再利用と概念的一貫性**を得る。

### 3.4 決定論・セーブ・タイムスケール

```
- 固定タイムステップ（例 1 tick = 固定 dt）でシミュレーションし、描画は補間で滑らかに
- 乱数は全てシード付きPRNG（xorshift/PCG）。乱数ストリームをシステム別に分離
  → 「決定論的シミュレーション」を保証
- セーブ = ワールド状態の完全シリアライズ（ECSのComponentストアをダンプ）
- 決定論 + 入力ログ で「リプレイ／タイムマシン」機能が安価に実現
```

> **なぜ決定論か**: (a) セーブ/ロードの完全一致、(b) バグ再現性（同シードで必ず再現）、
> (c) 後述 Chronicle の「なぜこうなったか」追跡、(d) マルチプレイ時の同期コスト削減（lockstep）。

### 3.5 ティック・パイプライン（システム実行順の確定）

```
1. InputApplySystem      … プレイヤーの場操作を反映
2. EcologyDiffusionSystem … 拡散方程式1ステップ
3. EconomyMatchSystem     … 市場約定・価格更新
4. NeedDecaySystem        … ニーズ自然減衰
5. PerceptionSystem       … 各住人の知覚（視界・噂）更新
6. UtilityDecisionSystem  … 行動決定（並列可）
7. ActionExecSystem       … 移動・採取・交流の実行
8. InteractionSystem      … 住人間相互作用→関係/記憶更新
9. EmotionUpdateSystem    … PAD更新・気分再計算
10. DisasterHazardSystem  … 状態依存ハザード判定
11. ChronicleSystem       … 重要イベントを物語ログに記録（§4.4）
12. LODManagerSystem      … LODの昇格/降格・遠方のreify/dehydrate
```

並列の境界はデータ依存で決める（例: 6 は読み取り中心で並列、8 は書き込み競合があるので決定論的順序で逐次 or 区画分割）。

## Step 4: UI/UX の哲学

### 4.1 基本哲学: 「Progressive Disclosure（漸進的開示）」と「Layered Information」

情報をすべて常時表示するのは初心者を殺し、隠しすぎると熟練者を退屈させる。
解は **「デフォルトは静か、要求に応じて深く」**。

```
3階層のズーム＝情報粒度の連動:
  Macro view  : 集落全体。色のヒートマップ（健康度/食料/治安）のみ。意思決定の"異常検知"用
  Meso view   : 地区。建物・物流フロー・人の動線アニメーション
  Micro view  : 個人。クリックで住人カード（感情/関係/記憶/今の思考の吹き出し）
```

> **なぜ Progressive Disclosure か**: 認知負荷理論（Cognitive Load Theory）。
> ワーキングメモリは同時に7±2チャンクしか保持できない。常時全情報＝過負荷＝離脱。
> 「今の意思決定に必要な情報だけ」を出し、ドリルダウンで詳細へ進ませることで学習コストを最小化する。

### 4.2 情報密度と視認性: 「オーバーレイ・トグル」方式

ベースマップは常にクリーンに保ち、情報は**半透明オーバーレイの重ね合わせ**で出す（SimCity の系譜）。

```
オーバーレイ例（ホットキーで瞬時トグル/重畳）:
  [水分] [汚染] [肥沃度] [治安] [幸福] [物流量] [所有権] [疫病リスク]
各オーバーレイは同一の色覚スキーム（青=低/赤=高）で統一し、学習を1回で済ませる
カラーユニバーサルデザイン対応（赤緑色覚に配慮し、色＋パターン併用）
```

これにより画面は「読みたい層だけ」を選択でき、**情報密度をプレイヤーが動的に制御**する。
密度を固定UIで決め打ちせず、ユーザーに委譲するのが視認性と深さの両立解。

### 4.3 熟練者向けの深さ: 「Inspect → Trace（因果追跡）」

熟練者の最大の欲求は **「なぜこうなった？」への回答**。あらゆる状態を**因果リンク付き**で開示する。

```
住人カードで「不機嫌」をクリック → 寄与している要因の内訳がスタック表示:
   不機嫌 (−24)
     ├ 配偶者の死を目撃 (−15, 3日前, salience 0.8)  ← クリックで該当イベントへジャンプ
     ├ 過密による不快 (−6)                          ← クリックで該当タイルへ
     └ 空腹 (−3)
```

価格・生態・災害も同様に「直近で効いた要因」を分解表示する。
**ブラックボックスを作らない**。すべての数値が「分解→源泉へジャンプ」可能。

> **なぜ因果追跡か**: §1.2 のカタルシスは「自分の因果寄与の明確さ」に比例する。
> プレイヤーが「自分の介入Aが→Bを引き起こし→Cという惨事/成功になった」を**追跡できて初めて**、
> 成功は誇りに、失敗は学習になる。追跡不能なゲームは「運ゲー」と化し創発の喜びが消える。
> これは熟練度の天井を上げる（システムを深く読む人ほど報われる）。

### 4.4 物語の可視化: 「Chronicle（年代記）」システム

創発したドラマを**プレイヤーが物語として認識**できなければ、せっかくの創発も「ただのログ」で終わる。
ChronicleSystem が高 salience イベントを**自然言語テンプレ＋関係文脈**で物語化して時系列に綴る。

```
"厳冬の3年目、食料を巡る諍いでアスラがベルンを突き飛ばした。
 この日からベルンはアスラを許さなかった——
 そして5年目の落盤事故の夜、ベルンは差し伸べるべき手を引っ込めた。"
```

これは LLM 生成でも、構造化テンプレ（イベント型 × 関係 × 感情 → 文）でも実装可能。
**「物語が勝手に生まれる」体験の最後のピース**＝創発を言語化して見せること。

> **なぜ Chronicle か**: Dwarf Fortress の伝説的な物語（Boatmurdered 等）は、
> プレイヤーがログを読んで**意味づけ（narrativize）**することで生まれた。
> その意味づけをシステムが補助すれば、物語性が万人に開かれる。創発の最後の増幅器。

### 4.5 学習コスト低減: 「Diegetic Tutorial（世界内チュートリアル）」と「Suggestion, not Command」

- チュートリアルは別画面のテキストではなく、**最初の住人が困りごとを吹き出しで訴える**（世界内＝diegetic）。
  プレイヤーは「説明を読む」のではなく「住人を助ける」過程で操作を学ぶ。
- アドバイザーは**命令せず示唆する**（"西区の食料が3日で尽きそうです"）。
  解法は与えない。プレイヤーの仮説形成（コアサイクルの第2フェーズ）を奪わない。

> **なぜ命令しないか**: 答えを与えるUIはコアサイクル（観察→仮説→介入）を短絡し、
> 発見の快感とカタルシスを破壊する。UIの役割は**問題の存在を気づかせること**であって、**解くこと**ではない。

## 5. 実装ロードマップ（MVP→拡張）

| フェーズ | 範囲 | 検証する仮説 |
|---|---|---|
| **P0: 縦の薄切り** | ECS基盤＋1タイル種＋10住人＋ニーズ/Utility AIのみ | 「自律行動が"生きてる"と感じるか」 |
| **P1: 心理層** | PAD感情＋関係グラフ＋記憶 | 「設計外の対人ドラマが出るか」 |
| **P2: 環境層** | 拡散方程式＋生態カスケード | 「空間的因果が学習可能か」 |
| **P3: 経済層** | 創発的価格＋技術DAG | 「最適解が固定化しないか」 |
| **P4: 統合＋カオス調整** | 3層結合＋状態依存ハザード | 「7:3の予測可能性に届くか」 |
| **P5: 体験層** | オーバーレイ/因果追跡/Chronicle | 「創発が物語として伝わるか」 |
| **P6: スケール** | LOD/HPA*/並列化 | 「1万体で60fpsを保つか」 |

各フェーズは**前フェーズの仮説が検証できてから次へ進む**（垂直スライス開発）。
創発系ゲームは「作ってみないと面白いか分からない」ため、**早期に遊べる薄切りを作り、観察で設計を反証する**のが唯一の正攻法。

## 付録B: この設計が「世界一」を狙える理由（要約）

1. **3層の閉ループ結合** → 設計外のドラマが構造的に発生する（偶然頼みでない創発）。
2. **間接統治（庭師メタファー）** → 自律性を殺さずプレイヤーに主体性を与える。
3. **状態依存ハザード（7:3）** → 予測可能性のスイートスポットで中毒性を最大化。
4. **因果追跡UI + Chronicle** → 創発を「自分の物語」として知覚させ、カタルシスを成立させる。
5. **ECS + LOD + 決定論** → 大規模・高拡張・再現可能を同時に満たす実装基盤。

> 創発は「足し算」ではなく「掛け算」で生まれる。
> 3つの単純で頑健なシステムを正しく結合すれば、組合せ爆発が無限の物語を生む。
> 本設計の全ての選択は、その掛け算を**意図的に・制御可能に**起こすためにある。
